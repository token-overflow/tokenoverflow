# Design: Snake Case Standardization

## Architecture Overview

### Goal

Standardize all project-owned names (directories, package names, binary names,
service names, container names, file names, skill names, agent names) to
snake_case across the TokenOverflow monorepo. The codebase currently mixes
kebab-case and snake_case for these names, creating inconsistency.

### Scope

This design covers **renaming and reference updates only**. No application code
logic, API contracts, database schemas, or runtime behavior changes. Every
command, workflow, and service must work identically before and after the
rename.

### What Is NOT Changing

The following items use kebab-case and will remain as-is because they follow
external conventions that are not ours to change:

| Item | Reason |
|------|--------|
| Tag format in MCP tools (`rust`, `tower-http`) | Stack Overflow convention; tags are kebab-case by design |
| MCP server instructions referencing tag format | Documents the tag convention above |
| AWS SSO profile names (`tokenoverflow-dev-admin`, etc.) | External AWS config owned by the identity provider |
| `docker-compose.yml` filename | Docker convention; the Docker CLI expects this exact name |
| `.claude-plugin` directory name | Claude Code plugin system convention |
| Claude Code skill/agent names and directories | Claude Code convention requires kebab-case for skill names, agent names, and their directory names |
| Third-party Cargo dependency names (`tower-http`, `diesel-async`, etc.) | Owned by upstream crate authors |
| AWS API property names (`awslogs-group`, `awslogs-region`, etc.) | AWS ECS API contract |
| Pre-commit hook IDs (`check-yaml`, `end-of-file-fixer`, etc.) | Owned by the pre-commit hook repos |
| External model names (`voyage-code-3`) | Owned by Voyage AI |
| `Cargo.lock` | Auto-regenerated by cargo |
| Design doc internal content | Historical records; only filenames change |

### Naming Convention Alternatives Considered

Before committing to snake_case, three conventions were evaluated:

| Convention | Example | Rust Ecosystem | Docker/OCI | Shell/Filesystem | Cargo Packages |
|------------|---------|----------------|------------|------------------|----------------|
| **snake_case** | `embedding_service` | Native (modules, functions, variables) | Works (underscores are valid in service/container names) | Works (no special chars) | Works (`name = "embedding_service"`) |
| **kebab-case** | `embedding-service` | Foreign (not idiomatic; `use embedding-service` is invalid) | Conventional (Docker docs use hyphens) | Works | Conventional (`name = "embedding-service"`, auto-mapped to `embedding_service` for `use`) |
| **camelCase** | `embeddingService` | Non-idiomatic | Non-conventional | Works but unusual | Non-conventional |

**Decision: snake_case.**

Rationale:

1. **Rust is the dominant language.** The entire backend is Rust. Rust's module
   system, function names, variable names, and file names all use snake_case.
   Having directory and package names match the language convention eliminates
   the mental translation between `embedding-service` (directory/package) and
   `embedding_service` (Rust module/import).

2. **Cargo auto-mapping creates confusion.** Cargo silently converts
   `embedding-service` to `embedding_service` for `use` statements. This means
   the package name in `Cargo.toml` differs from how you reference it in Rust
   code. Using snake_case everywhere removes this discrepancy.

3. **Docker supports underscores.** While Docker documentation tends to use
   hyphens, underscores are fully supported in service names, container names,
   and image names. There is no technical limitation.

4. **Consistency is the goal.** The current mix of kebab-case and snake_case is
   the problem. Picking one convention and applying it uniformly is more
   important than which convention is picked. snake_case wins because the
   codebase is Rust-first.

Trade-off: Docker Compose service names with underscores look slightly unusual
to developers accustomed to the Docker convention of hyphens. This is cosmetic
and does not affect functionality.

### Infrastructure Impact

Renaming the Terraform RDS identifier from `main-prod` to `main_prod` will
cause Terraform to destroy and recreate the RDS instance on next apply. This
is acceptable because the project is in early development with no production
data. The user has explicitly confirmed this is fine.

### Doc Generator Script (`scripts/src/docs.sh`)

The `create_doc` function must be updated to:

1. **Validate** that the `name` parameter is snake_case (reject hyphens).
2. **Convert** snake_case to Title Case for the `{{name}}` template placeholder.

The filename stays snake_case (e.g., `2026_02_20_snake_case_standardization.md`),
but the heading inside the file becomes Title Case (e.g.,
`# Design: Snake Case Standardization`).

**Before:**

```bash
function create_doc() {
  local type="$1"
  local name="$2"
  # ...
  local filename="${date_prefix}_${name}.md"
  mkdir -p "$target_dir"
  sed "s/{{name}}/${name}/g" "$template" >"${target_dir}/${filename}"
  echo "Created ${target_dir}/${filename}"
}
```

**After:**

```bash
function create_doc() {
  local type="$1"
  local name="$2"

  if [[ -z "$type" || -z "$name" ]]; then
    echo "Usage: create_doc <brief|prd|design> <name>" >&2
    return 1
  fi

  if [[ "$name" =~ - ]]; then
    echo "Error: name must be snake_case (no hyphens). Got: $name" >&2
    return 1
  fi

  case "$type" in
    brief | prd | design) ;;
    *)
      echo "Unknown type: $type (expected: brief|prd|design)" >&2
      return 1
      ;;
  esac

  local template="docs/templates/${type}.md"
  local target_dir="docs/${type}"
  local date_prefix
  date_prefix=$(date +%Y_%m_%d)
  local filename="${date_prefix}_${name}.md"

  # Convert snake_case to Title Case for the document heading
  local title
  title=$(echo "$name" | tr '_' ' ' | awk '{for(i=1;i<=NF;i++) $i=toupper(substr($i,1,1)) substr($i,2)}1')

  mkdir -p "$target_dir"
  sed "s/{{name}}/${title}/g" "$template" >"${target_dir}/${filename}"
  echo "Created ${target_dir}/${filename}"
}
```

**Example:** `create_doc design snake_case_standardization` produces:
- Filename: `docs/design/2026_02_20_snake_case_standardization.md`
- Heading: `# Design: Snake Case Standardization`

The templates (`docs/templates/*.md`) remain unchanged — they still use
`{{name}}` as the placeholder. The conversion happens in `create_doc`.

```dockerfile
# =============================================================================
# Stage 1: Chef - Base image with cargo-chef installed
# =============================================================================
FROM rust:1.93-slim AS chef
WORKDIR /app

# Install build dependencies
RUN apt-get update && apt-get install -y \
    pkg-config \
    libssl-dev \
    g++ \
    && rm -rf /var/lib/apt/lists/*

RUN cargo install cargo-chef

# =============================================================================
# Stage 2: Planner - Generate dependency recipe
# =============================================================================
FROM chef AS planner

# Copy workspace manifest, lockfile, and only this service's source
COPY Cargo.toml ./Cargo.toml
COPY Cargo.lock ./Cargo.lock
COPY apps/embedding_service ./apps/embedding_service

# Replace workspace manifest with single-member workspace
RUN printf '[workspace]\nresolver = "2"\nmembers = ["apps/embedding_service"]\n' > Cargo.toml

# Generate recipe.json for dependency caching
RUN cargo chef prepare --recipe-path recipe.json

# =============================================================================
# Stage 3: Builder - Compile dependencies then application
# =============================================================================
FROM chef AS builder

# Copy recipe and build dependencies (cached layer)
COPY --from=planner /app/recipe.json recipe.json
RUN cargo chef cook --release --recipe-path recipe.json

# Copy workspace manifest, lockfile, and only this service's source
COPY Cargo.toml ./Cargo.toml
COPY Cargo.lock ./Cargo.lock
COPY apps/embedding_service ./apps/embedding_service

# Replace workspace manifest with single-member workspace
RUN printf '[workspace]\nresolver = "2"\nmembers = ["apps/embedding_service"]\n' > Cargo.toml

# Build the binary
RUN cargo build --release -p embedding_service

# =============================================================================
# Stage 4: Runtime - Minimal production image
# =============================================================================
FROM debian:bookworm-slim AS runtime

# Install runtime dependencies
RUN apt-get update && apt-get install -y \
    ca-certificates \
    libssl3 \
    curl \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# Copy the binary
COPY --from=builder /app/target/release/embedding_service /app/embedding_service

# Create cache directory for model (will be downloaded on first run)
RUN mkdir -p /app/.fastembed_cache

# Create non-root user for security
RUN useradd -r -u 1001 appuser && chown -R appuser:appuser /app
USER appuser

EXPOSE 8080

ENV HOST=0.0.0.0
ENV PORT=8080
ENV RUST_LOG=info

HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1

CMD ["/app/embedding_service"]
```

## Edge Cases & Constraints

### 1. Docker DNS resolution with underscores

**Risk:** Docker Compose creates a DNS entry for each service name. Underscores
in DNS names are technically RFC-noncompliant (RFC 952), though Docker's
internal DNS resolver handles them without issue.

**Mitigation:** Docker Compose uses its own embedded DNS server for
inter-container resolution. Underscored service names like
`embedding_service` are fully supported and widely used. This is not a
practical concern.

### 2. Cargo binary name change

**Risk:** Changing the binary name from `embedding-service` to
`embedding_service` changes the compiled artifact name in `target/release/`.
Any scripts, Dockerfiles, or deployment configs that reference the binary name
must be updated.

**Mitigation:** This design doc explicitly lists the Dockerfile as a file that
must be updated (Step 2.5). The API binary name (`tokenoverflow`) is not
changing. The only affected binaries are `embedding_service` and `so_tag_sync`.

### 3. Terraform state drift

**Risk:** Renaming the IAM policy (`api-secrets-read` to `api_secrets_read`)
and RDS identifier (`main-prod` to `main_prod`) will cause Terraform to
destroy and recreate these resources.

**Mitigation:** The user has confirmed this is acceptable. The project has no
production data. After running `terragrunt run --all apply`, the resources
will be recreated with the new names. The Terraform state will be consistent.

### 4. All changes must be in one commit

**Risk:** If directory renames and file content updates are done in separate
commits, intermediate states will be broken. For example, renaming
`apps/embedding-service/` to `apps/embedding_service/` without updating
`Cargo.toml` members will break `cargo build`.

**Mitigation:** All Phase 1 (renames) and Phase 2 (content updates) steps must
be staged together and committed as a single commit. Pre-commit hooks should be
skipped during intermediate work using `git commit --no-verify` only if
needed, but the final commit must pass all hooks.

### 5. Existing Docker volumes and containers

**Risk:** After changing container names (e.g., `tokenoverflow-db` to
`tokenoverflow_db`), `docker compose up` will create new containers instead of
reusing old ones. Old containers will be orphaned.

**Mitigation:** Run `docker compose down` before the rename to clean up old
containers. After the rename, `docker compose up -d` will create containers
with the new names. Docker volumes are not affected by container name changes
-- `postgres_data` and `embedding_model_cache` volume names are not changing.

### 6. Design doc content is NOT updated

**Risk:** Existing design docs contain references to old names (e.g.,
`apps/embedding-service/` in the monorepo restructure doc). These are
historical records.

**Mitigation:** Only the filenames of design docs are renamed. Internal content
is preserved as-is. Design docs are snapshots of the codebase at the time they
were written and should not be retroactively edited.

### 7. `MEMORY.md` in user's private Claude config

**Risk:** The user's `MEMORY.md` references old names like
`apps/embedding-service/`.

**Mitigation:** The user should update their `MEMORY.md` after the rename. This
is outside the scope of codebase changes but should be noted as a
post-migration step.
