# Design: fck-nat

## Architecture Overview

### Goal

Deploy a cost-effective NAT solution for the production VPC's private subnets
using fck-nat, an open-source NAT instance AMI, orchestrated by the official
RaJiska Terraform module. This replaces the deferred "enable AWS Managed NAT
Gateway" plan from the VPC design document and enables private subnet workloads
(e.g., ECS tasks, EC2 instances) to reach the internet for pulling container
images, calling external APIs (Voyage AI, etc.), and other outbound traffic.

### Scope

This design covers:

- fck-nat deployment using the `RaJiska/fck-nat/aws` Terraform module (v1.4.0)
- High-availability mode via Auto Scaling Group (single instance, auto-replace)
- Floating internal ENI for consistent private IP across instance replacements
- Route table updates pointing private subnets to the fck-nat ENI
- Security group configuration
- Cost analysis and instance type selection

This design does NOT cover:

- Elastic IP assignment (not required; the fck-nat module automatically
  allocates and associates an EIP to the NAT instance's ENI, so
  `map_public_ip_on_launch = false` on the public subnet is not a problem)
- SSH access to the NAT instance (disabled for security)
- CloudWatch agent (excluded due to cost; see Decision section)
- Dev environment NAT (deferred, but directory structure supports it)
- VPC Flow Logs (separate concern)
- Custom CloudWatch alarms (deferred)
- Spot instances (explicitly excluded per requirements)

### Why fck-nat Instead of AWS Managed NAT Gateway

The VPC design document deferred NAT Gateway with a note that enabling it would
require only changing two lines in the VPC module. However, AWS Managed NAT
Gateway costs approximately $32.85/month (hourly charge) plus $0.045/GB in data
processing fees. For an early-stage MVP with minimal traffic, this is
unnecessarily expensive.

fck-nat is an open-source NAT instance AMI that provides the same outbound NAT
functionality at a fraction of the cost. It uses a standard EC2 instance with
IP forwarding enabled and iptables rules configured automatically by the AMI.

| Criteria | AWS Managed NAT Gateway | fck-nat (t4g.nano) |
|----------|------------------------|--------------------|
| Monthly base cost | ~$32.85 | ~$3.07 |
| Data processing fee | $0.045/GB | $0 (standard EC2 networking) |
| Availability | AWS-managed, multi-AZ capable | ASG auto-replaces failed instances |
| Bandwidth | Up to 100 Gbps | Up to 5 Gbps burst (t4g.nano) |
| Operational overhead | Zero | Minimal (ASG handles replacement) |
| Estimated monthly cost at 10 GB traffic | ~$33.30 | ~$3.07 |
| Estimated monthly cost at 100 GB traffic | ~$37.35 | ~$3.07 |

For the MVP stage, fck-nat provides more than sufficient bandwidth and
reliability at roughly 10% of the cost. If traffic grows to the point where
5 Gbps becomes a bottleneck or operational overhead becomes a concern, migrating
to a managed NAT Gateway is a straightforward change.

### CloudWatch Agent Decision

The fck-nat module supports an optional CloudWatch agent that reports metrics
similar to the managed NAT Gateway (bytes in/out, packets, connection tracking,
etc.). However, the fck-nat documentation explicitly warns that the CloudWatch
agent configuration costs approximately $17/month in custom metric charges
(excluding free tier).

| Option | Monthly Cost | Metrics Available |
|--------|-------------|-------------------|
| A. CloudWatch agent disabled | $0 | None (rely on EC2 basic monitoring) |
| B. CloudWatch agent enabled | ~$17 | Full NAT Gateway-like metrics |

**Decision: Option A (disabled).** At $17/month, the CloudWatch agent would
cost more than 5x the NAT instance itself ($3.07/month). EC2 basic monitoring
(CPU, network in/out, status checks) is included for free and is sufficient
for an MVP. If detailed NAT metrics become necessary, the agent can be enabled
later by setting `use_cloudwatch_agent = true`.

### Instance Type Decision

The requirement specifies the cheapest possible instance type. In `us-east-1`,
the ARM-based (Graviton) instance types offer the best price-performance ratio,
and fck-nat defaults to an ARM AMI.

| Instance Type | Architecture | vCPUs | Memory | Network | Monthly Cost (on-demand) |
|---------------|-------------|-------|--------|---------|-------------------------|
| t4g.nano | ARM (Graviton2) | 2 | 0.5 GB | Up to 5 Gbps | $3.07 |
| t4g.micro | ARM (Graviton2) | 2 | 1 GB | Up to 5 Gbps | $6.05 |
| t4g.small | ARM (Graviton2) | 2 | 2 GB | Up to 5 Gbps | $12.26 |
| t3.nano | x86_64 | 2 | 0.5 GB | Up to 5 Gbps | $3.80 |

**Decision: t4g.nano.** This is the cheapest instance type available on AWS.
The fck-nat AMI is optimized for Graviton processors and the 0.5 GB of memory
is more than sufficient for NAT forwarding (iptables rules use negligible
memory). The 5 Gbps burst bandwidth matches the managed NAT Gateway's base
throughput. The fck-nat module defaults to `t4g.micro`, so we override this
to `t4g.nano`.

### High-Availability Mode

The fck-nat module supports two deployment modes:

| Mode | How It Works | Failover | Cost |
|------|-------------|----------|------|
| Non-HA (`ha_mode = false`) | Single EC2 instance, no ASG | Manual: must re-create instance | Same instance cost |
| HA (`ha_mode = true`) | ASG with min=max=desired=1 | Automatic: ASG replaces unhealthy instance in ~2-3 min | Same instance cost + no extra charge for ASG |

**Decision: HA mode enabled.** The ASG wrapper costs nothing extra (ASG itself
is free; you only pay for the EC2 instance). When the NAT instance fails a
health check or is terminated, the ASG automatically launches a replacement.
The new instance attaches the same floating ENI, so the private IP and route
table entries remain valid without any manual intervention.

### ENI Strategy

In HA mode, fck-nat creates a dedicated internal ENI in the specified subnet.
This ENI:

1. Has source/destination checking disabled (required for NAT)
2. Gets a private IP from the public subnet's CIDR range
3. Is automatically attached to new instances launched by the ASG
4. Persists across instance replacements (the ENI is an independent resource)

The route table entries point to this ENI (not the instance), so when the ASG
replaces an instance, the routes remain valid as soon as the new instance
attaches the ENI.

The module handles ENI creation and attachment automatically. No manual ENI
management is required.

### Network Architecture

```text
              Internet
                 |
            [IGW] (already exists)
                 |
  +--------------+-----------------------------+
  |         VPC 10.0.0.0/16                    |
  |                                            |
  |  Tier 1: Public Subnets                    |
  |  +------------------+  +----------------+ |
  |  | 10.0.0.0/24      |  | 10.0.1.0/24    | |
  |  | us-east-1a       |  | us-east-1b     | |
  |  |                  |  |                 | |
  |  | [fck-nat]        |  |                 | |
  |  | (t4g.nano, ASG)  |  |                 | |
  |  | ENI: 10.0.0.x    |  |                 | |
  |  +--------+---------+  +----------------+ |
  |           |                                |
  |           | Route: 0.0.0.0/0 -> ENI        |
  |           |                                |
  |  Tier 2: Private Subnets                   |
  |  +------------------+  +----------------+ |
  |  | 10.0.10.0/24     |  | 10.0.11.0/24   | |
  |  | us-east-1a       |  | us-east-1b     | |
  |  | (App servers)    |  | (App servers)  | |
  |  +------------------+  +----------------+ |
  |                                            |
  |  Tier 3: Isolated Subnets                  |
  |  +------------------+  +----------------+ |
  |  | 10.0.20.0/24     |  | 10.0.21.0/24   | |
  |  | us-east-1a       |  | us-east-1b     | |
  |  | (RDS)            |  |                 | |
  |  +------------------+  +----------------+ |
  +--------------------------------------------+
```

**Traffic flow:**

1. App server in private subnet (10.0.10.x) sends a packet to the internet
2. Private route table has `0.0.0.0/0 -> eni-xxx` (fck-nat's internal ENI)
3. Packet reaches fck-nat instance via the ENI
4. fck-nat performs SNAT (source NAT) and forwards the packet via the public
   subnet's IGW route
5. Response follows the reverse path

**Key points:**

- fck-nat is deployed in the first public subnet (us-east-1a) where it gets
  a public IP via the IGW
- Both private route tables (us-east-1a and us-east-1b) point to the same
  fck-nat ENI, so both AZs share a single NAT instance
- Cross-AZ traffic (from 10.0.11.0/24 to fck-nat in us-east-1a) incurs a
  small AWS cross-AZ data transfer charge ($0.01/GB), which is negligible at
  MVP traffic levels
- Isolated subnets (tier 3) are NOT routed through fck-nat; they remain
  fully isolated by design

### Module Strategy

Consistent with the VPC and RDS designs, a wrapper module pattern is used:

```text
infra/terraform/
  modules/
    nat/                          # new wrapper module
      main.tf                     # fck-nat community module call
      variables.tf
      outputs.tf
  live/
    prod/
      nat/                        # new Terragrunt unit
        terragrunt.hcl
```

The wrapper module encapsulates the community module call with
project-specific defaults and provides a stable interface for Terragrunt
units.

### Terragrunt Dependency

The fck-nat unit depends on the VPC unit to obtain:

- `vpc_id` (for security group placement)
- `public_subnet_ids` (for NAT instance placement)
- `private_route_table_ids` (for route table updates)

This is expressed as a Terragrunt `dependency` block, following the same
pattern as the RDS unit.

## Interfaces

This section documents every file that will be created or modified, and the
contracts between them.

### New Files

#### `infra/terraform/modules/nat/variables.tf`

```hcl
variable "env_name" {
  description = "Environment name (e.g., prod, dev). Used for resource naming and tagging."
  type        = string
}

variable "vpc_id" {
  description = "ID of the VPC where the NAT instance will be deployed."
  type        = string
}

variable "subnet_id" {
  description = "ID of the public subnet where the NAT instance will be placed."
  type        = string
}

variable "route_tables_ids" {
  description = "Map of route table names to IDs. These route tables will have 0.0.0.0/0 pointed to the fck-nat ENI."
  type        = map(string)
}

variable "instance_type" {
  description = "EC2 instance type for the NAT instance."
  type        = string
  default     = "t4g.nano"
}
```

#### `infra/terraform/modules/nat/main.tf`

```hcl
module "fck_nat" {
  source  = "RaJiska/fck-nat/aws"
  version = "1.4.0"

  name      = "fck-nat-${var.env_name}"
  vpc_id    = var.vpc_id
  subnet_id = var.subnet_id

  # HA mode: NAT instance wrapped in ASG (min=max=desired=1)
  # ASG auto-replaces unhealthy instances. Floating ENI persists across replacements.
  ha_mode = true

  # Instance configuration
  instance_type      = var.instance_type
  use_spot_instances = false

  # Route table updates: point 0.0.0.0/0 to the fck-nat ENI
  update_route_tables = true
  route_tables_ids    = var.route_tables_ids

  # CloudWatch agent: disabled ($17/month is too expensive for a $3/month instance)
  use_cloudwatch_agent = false

  # SSM agent: enabled for remote access without SSH
  attach_ssm_policy = true

  # SSH: disabled (use SSM Session Manager instead)
  use_ssh = false

  # EBS encryption: enabled (uses default AWS managed key)
  encryption = true

  # Tags
  tags = {
    Environment = var.env_name
    ManagedBy   = "opentofu"
  }
}
```

#### `infra/terraform/modules/nat/outputs.tf`

```hcl
output "eni_id" {
  description = "The ID of the static ENI used by the fck-nat instance."
  value       = module.fck_nat.eni_id
}

output "security_group_ids" {
  description = "List of security group IDs used by fck-nat ENIs."
  value       = module.fck_nat.security_group_ids
}

output "role_arn" {
  description = "The ARN of the IAM role used by the fck-nat instance profile."
  value       = module.fck_nat.role_arn
}

output "autoscaling_group_arn" {
  description = "The ARN of the autoscaling group managing the fck-nat instance."
  value       = module.fck_nat.autoscaling_group_arn
}

output "launch_template_id" {
  description = "The ID of the launch template used to spawn fck-nat instances."
  value       = module.fck_nat.launch_template_id
}
```

#### `infra/terraform/live/prod/nat/terragrunt.hcl`

```hcl
include "root" {
  path = find_in_parent_folders("root.hcl")
}

terraform {
  source = "../../../modules/nat"
}

dependency "vpc" {
  config_path = "../vpc"
}

inputs = {
  env_name  = "prod"
  vpc_id    = dependency.vpc.outputs.vpc_id
  subnet_id = dependency.vpc.outputs.public_subnet_ids[0]

  route_tables_ids = {
    for idx, id in dependency.vpc.outputs.private_route_table_ids :
    "private-${idx}" => id
  }

  # All other inputs use module defaults:
  # instance_type = "t4g.nano"
}
```

The fck-nat module manages routes independently via the `update_route_tables`
variable, which creates `aws_route` resources pointing `0.0.0.0/0` to the
fck-nat ENI in the specified route tables.

## Logic

This section defines the exact sequence of operations to implement the fck-nat
infrastructure.

### Phase 1: Create the fck-nat wrapper module

**Step 1.1:** Create the module directory:

```bash
mkdir -p infra/terraform/modules/nat
```

**Step 1.2:** Create `infra/terraform/modules/nat/variables.tf` with the
content defined in the Interfaces section.

**Step 1.3:** Create `infra/terraform/modules/nat/main.tf` with the content
defined in the Interfaces section.

**Step 1.4:** Create `infra/terraform/modules/nat/outputs.tf` with the
content defined in the Interfaces section.

**Step 1.5:** Validate with TFLint:

```bash
cd infra/terraform/modules/nat
tflint --config="$(pwd)/../../.tflint.hcl" --init
tflint --config="$(pwd)/../../.tflint.hcl"
```

### Phase 2: Create the prod Terragrunt unit

**Step 2.1:** Create the unit directory:

```bash
mkdir -p infra/terraform/live/prod/nat
```

**Step 2.2:** Create `infra/terraform/live/prod/nat/terragrunt.hcl` with
the content defined in the Interfaces section.

### Phase 3: Deploy fck-nat to production

**Step 3.1:** Log in to the prod AWS account:

```bash
aws sso login --profile tokenoverflow-prod-admin
```

**Step 3.2:** Initialize the prod fck-nat unit:

```bash
cd infra/terraform/live/prod/nat
terragrunt init
```

**Step 3.3:** Review the plan:

```bash
terragrunt plan
```

Expected resources to be created:

- 1 `aws_network_interface` (internal ENI, source/dest check disabled)
- 1 `aws_security_group` (allows VPC ingress, all egress)
- 1 `aws_launch_template` (t4g.nano, fck-nat AMI, instance profile)
- 1 `aws_autoscaling_group` (min=max=desired=1)
- 1 `aws_iam_role` + 1 `aws_iam_instance_profile` (ENI attach + SSM)
- 1-2 `aws_route` (one per private route table, `0.0.0.0/0` -> ENI)
- IAM policies for ENI attachment and SSM

The plan should NOT show any changes to VPC or RDS resources.

**Step 3.4:** Apply:

```bash
terragrunt apply
```

Note: The ASG will launch a t4g.nano instance which typically starts within
1-2 minutes.

**Step 3.5:** Verify the instance is running:

```bash
aws autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names "fck-nat-prod" \
  --profile tokenoverflow-prod-admin \
  --region us-east-1 \
  --query 'AutoScalingGroups[0].Instances[0].{Id:InstanceId,State:LifecycleState,Health:HealthStatus}'
```

**Step 3.6:** Verify route tables are updated:

```bash
VPC_ID=$(cd infra/terraform/live/prod/vpc && terragrunt output -raw vpc_id)

aws ec2 describe-route-tables \
  --filters "Name=vpc-id,Values=$VPC_ID" \
  --query 'RouteTables[*].{ID:RouteTableId,Routes:Routes[*].{Dest:DestinationCidrBlock,Target:NetworkInterfaceId},Tags:Tags[?Key==`Name`].Value|[0]}' \
  --profile tokenoverflow-prod-admin \
  --region us-east-1 \
  --output table
```

**Step 3.7:** Commit:

```bash
git add infra/terraform/modules/nat/
git add infra/terraform/live/prod/nat/
git commit -m "infra: add fck-nat for cost-effective NAT in prod VPC"
```

## Edge Cases & Constraints

### 1. Single NAT instance serves both AZs

**Risk:** All outbound internet traffic from both private subnets
(us-east-1a and us-east-1b) routes through a single fck-nat instance in
us-east-1a. If the instance fails, both AZs lose outbound connectivity
until the ASG replaces it (approximately 2-3 minutes).

**Mitigation:** This is an accepted trade-off for cost savings. A managed NAT
Gateway also has a single point of failure when deployed in one AZ. For true
multi-AZ NAT redundancy, you would need two NAT instances (one per AZ) at
double the cost. At MVP traffic levels, the 2-3 minute recovery time from
the ASG is acceptable.

### 2. Cross-AZ data transfer charges

**Risk:** Traffic from the us-east-1b private subnet (10.0.11.0/24) to the
fck-nat instance in us-east-1a incurs AWS cross-AZ data transfer charges of
$0.01/GB in each direction.

**Mitigation:** At MVP traffic levels, this cost is negligible (e.g., 10 GB
cross-AZ traffic costs $0.20/month). If traffic grows significantly and
cross-AZ costs become material, deploy a second fck-nat instance in
us-east-1b with its own route table entry.

### 3. t4g.nano CPU credit exhaustion

**Risk:** The t4g.nano instance type uses burstable CPU credits. Under
sustained high CPU load (unlikely for NAT forwarding, which is I/O-bound),
CPU credits could be exhausted, throttling the instance to baseline
performance.

**Mitigation:** NAT forwarding is handled by the kernel's netfilter/iptables
subsystem and is primarily network-bound, not CPU-bound. The t4g.nano's
baseline CPU performance is sufficient for typical NAT workloads. If CPU
credit exhaustion occurs, upgrade to t4g.micro (2x cost) which provides more
CPU credits.

### 4. ENI attachment during ASG replacement

**Risk:** When the ASG replaces the NAT instance, there is a brief window
(seconds) between the new instance launching and the ENI being attached where
outbound traffic will fail.

**Mitigation:** The fck-nat AMI includes a startup script that attaches the
ENI as one of its first actions. The total downtime during replacement is
typically 2-3 minutes (instance launch + ENI attachment). Applications should
implement retry logic for outbound connections, which is standard practice.

### 5. Route table conflict with VPC module

**Risk:** If someone later sets `enable_nat_gateway = true` in the VPC module,
both the managed NAT Gateway and fck-nat would attempt to create a
`0.0.0.0/0` route in the private route tables, causing a Terraform conflict.

**Mitigation:** The VPC module's `enable_nat_gateway` is explicitly set to
`false` and documented as such. The fck-nat module manages the `0.0.0.0/0`
route independently. Before enabling a managed NAT Gateway, the fck-nat unit
must be destroyed first. This is documented in the Documentation Changes
section.

### 6. fck-nat AMI updates

**Risk:** The fck-nat module uses the latest AMI by default
(`ami_id = null`). When the fck-nat project publishes a new AMI, the next
`terraform plan` will show a change to replace the launch template.

**Mitigation:** AMI updates are expected and generally safe since fck-nat AMIs
are backward compatible. The ASG performs a rolling replacement, so the
downtime window is the same as any instance replacement (~2-3 minutes). To
pin a specific AMI version, set the `ami_id` variable explicitly.

### 7. Security group allows all VPC ingress

**Risk:** The fck-nat module's default security group allows all inbound
traffic from within the VPC CIDR. This is broader than strictly necessary.

**Mitigation:** This is the fck-nat module's default and is required for the
NAT instance to accept traffic from all private subnets. The instance is in
a public subnet but has no SSH access enabled and no services listening
except for the NAT forwarding. The attack surface is minimal because the
instance only forwards packets; it does not process application-layer
traffic.

### 8. Isolated subnets remain isolated

**Risk:** Someone might expect that deploying fck-nat gives the database
subnets internet access.

**Mitigation:** The `route_tables_ids` input only includes private route
table IDs, not database route table IDs. The isolated subnets' route tables
are not modified and continue to have only the local VPC route. This is
intentional and documented.
